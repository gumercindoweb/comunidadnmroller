## 2026-07-20 — Panel de ventas: confirmar pagos en efectivo desde turnos Calendly

- **Problema:** los turnos del beneficio "pago en efectivo" (showroom) se agendan por Calendly, pero no había forma de saber con certeza quién pagó al final, con qué plan, ni un registro de esa conversión. Tampoco entraban a ninguna secuencia de GetResponse (a diferencia del flujo de Mercado Pago + comprobante).
- **Hallazgo clave que simplificó el diseño:** Calendly ya captura el plan elegido como respuesta a una pregunta fija del formulario ("¿Cuál es el plan que querés pagar/comprar?"), además de DNI y WhatsApp — confirmado con una llamada real a la API. No hizo falta inventar un puente de UTM.
- **Decisión:** los turnos se leen en vivo de la API de Calendly (no se sincroniza un espejo completo); solo se persiste en Supabase (`pagos_efectivo`) la conversión que el vendedor confirma: pagado / no se presentó / vino y no pagó, con el plan editable. Al confirmar "pagado", se da de alta el contacto en una lista EXCLUSIVA de GetResponse (mismo patrón que la lista de socios SportClub).
- **Login del vendedor:** Supabase Auth (usuarios creados a mano, sin self-signup) — se descartó una contraseña fija en el frontend por ser trivialmente saltable (el código del sitio es público).
- **Implementado:** migración `pagos_efectivo`, Edge Functions `listar-turnos-efectivo` y `confirmar-pago-efectivo`, ruta `/panel-ventas` (+ `/panel-ventas/login`). Todo commiteado y pusheado a `main`.
- **Pendiente (pasos manuales, no automatizables desde acá):** crear el Personal Access Token de Calendly, crear la campaign exclusiva en GetResponse, crear el/los usuario(s) de Supabase Auth, aplicar la migración y cargar los secrets en el proyecto de producción `bosutrnpmpjxylcgjmbt`, y deployar ambas Edge Functions **sin** `--no-verify-jwt`. Sin estos pasos el panel no funciona todavía en producción. → **RESUELTO el 21-jul, ver la entrada de esa fecha más abajo.**
- **Probado con datos simulados** (`/panel-ventas/demo`, ruta temporal no commiteada) antes de tocar producción — flujo completo verificado: confirmar cambia el resumen/tasa de conversión en vivo.
- **Iteración 2 (mismo día):** todos los vendedores van a compartir un único login de Supabase Auth, así que se agregó un campo `vendedor` (texto libre, cargado a mano y recordado por `localStorage` en cada dispositivo) para saber quién gestionó cada turno — el email de la sesión (`confirmado_por`) no alcanza para distinguir personas. Además, GetResponse pasó de recibir solo los "pagados" a recibir los 3 estados (todos entran a la misma lista), con una etiqueta auto-creada por nombre ("Pagó en efectivo" / "No pagó en efectivo") para poder armar secuencias de mail distintas para quien compró vs. quien no. **Límite explícito aclarado al usuario:** Claude no carga API keys/tokens en Supabase ni deploya a producción, ni con autorización explícita — es una acción prohibida sin excepción, más allá de que el propio CLAUDE.md ya documentaba el bloqueo técnico del llavero de macOS (error #15). Se preparan los comandos exactos para que el usuario los pegue.
- **Iteración 2.1:** el campo `vendedor` pasó de texto libre a una lista fija (Fruti, Ana, Martu, Rama, Nico, Vladi) + "Otro" para casos no contemplados, para evitar errores de tipeo/variantes del mismo nombre.
- **Iteración 3 (mismo día):** dos ajustes más. (1) Calendly separa `first_name`/`last_name` del invitado (confirmado con la API real) — se usa eso para mandar nombre y apellido a GetResponse como 2 custom fields separados ("Nombre"/"Apellido", auto-creados si no existen) en vez de un solo campo combinado; el `name` completo se sigue mandando también, solo para que el contacto se identifique en el listado de GetResponse. (2) Se agregaron 2 fechas visibles en la tabla: cuándo se agendó el turno (de `invitee.created_at` de Calendly) y cuándo se confirmó el resultado (de `updated_at` en `pagos_efectivo`, que se pisa solo cada vez que el vendedor cambia el estado — sin lógica nueva, ya lo hacía el trigger existente). **Nota de proceso:** los nombres ficticios "Rocío"/"Facundo" que aparecieron en una captura eran datos de la vista simulada (`/panel-ventas/demo`), no de ningún sistema real — se reemplazaron por nombres reales de la lista fija para no repetir la confusión.
- **Iteración 4 (mismo día):** el campo de plan pasó de texto libre a un catálogo real, sacado de `PricingSection.tsx` y `PricingAlquilerSection.tsx` (8 planes: mensuales, trimestrales y clases+alquiler), cada uno con su precio de pago en efectivo — tabla nueva `planes_efectivo`, editable sin redeploy desde `/panel-ventas/configuracion`. Detalle no trivial: "Clase Única" se repite con precios distintos según el catálogo ($30.000 mensual vs. $35.000 alquiler) — confirmado con el usuario que la diferencia es el servicio de alquiler incluido, no un error de carga — así que la clave única es (categoría, plan_slug), no el nombre solo. El select se precarga con el plan que el cliente preguntó en Calendly, matcheado por nombre — el vendedor lo corrige si hace falta.
- **Iteración 5 (mismo día):** el panel ahora reacciona a reprogramaciones reales de Calendly en vez de quedarse con un estado viejo — si `invitee.rescheduled` viene en true, el turno pisa cualquier confirmación previa (menos "pagado") y pasa a "Reprogramó"; el turno nuevo generado por la reprogramación aparece por su cuenta como pendiente. Se sumaron tooltips explicando cada métrica del resumen, un selector de rango (30/60/90/180 días) para no perder reprogramaciones que caigan fuera de la ventana por defecto, y un botón "Actualizar" con etiqueta visible. Además, GetResponse ahora elige la lista según la categoría del plan (secret nuevo opcional `GETRESPONSE_CAMPAIGN_ID_PAGO_EFECTIVO_ALQUILER`, con fallback a la general), reflejando que alquiler y NM son públicos distintos.
- **Decisión de arquitectura (aclarada por el usuario, 20-jul-2026):** el panel de ventas es conceptualmente un **proyecto independiente** del sitio — solo administra la confirmación de pagos en efectivo vía turnos de Calendly, y **no tiene que sincronizarse con Lovable** en ningún momento (Lovable gestiona el sitio público, no esta herramienta interna). Está interconectado (comparte el mismo proyecto Supabase `bosutrnpmpjxylcgjmbt` como backend) pero no es lo mismo que el sitio. **Pendiente a futuro (no ahora):** separar todo el código del panel (páginas `PanelVentas*`, componentes `panel-ventas/`, Edge Functions `listar-turnos-efectivo` y `confirmar-pago-efectivo`, migración `pagos_efectivo`/`planes_efectivo`) a su propio repo bajo la cuenta `escuelanmroller`, manteniendo el mismo proyecto Supabase como base de datos compartida (las funciones y tablas no están atadas a este repo en particular). Falta pensar cómo impacta el corte en la base de datos/Edge Functions ya deployadas. Hasta que se haga ese corte, seguir el flujo de commit+push a `main` de este repo como con cualquier otro cambio (según lo instruye el CLAUDE.md del proyecto) — la separación es un trabajo aparte, no algo a resolver ahora.
